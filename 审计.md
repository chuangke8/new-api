## 图片生成页面

请求入口：`web/default/src/features/workspace-generation/index.tsx`

接口入口：`/api/workspace/image/generations`

后端入口：`controller/workspace_image.go`

中转路径：`GenerateWorkspaceImage` -> `middleware.Distribute()` -> `Relay(OpenAIImage)` -> `relay.ImageHelper()` -> `service.PostTextConsumeQuota()`

参数逻辑：

- 模型分组来自 `/api/workspace/image/models` 返回的 `category_id/category_alias/category_name`。
- 模型渠道显示优先级为 `model_alias/display_name/model`，提交给上游仍使用真实 `model`。
- 提示词 `prompt` 必填。
- 负面提示词受 `negative_prompt` 功能控制影响，关闭时前端清空并不提交。
- 参考图受 `reference_image_upload` 控制，上传后转为 data URL，通过 `image` 字段提交。
- 尺寸、比例、风格、质量分别受渠道功能控制影响。当前图片真正提交给上游的是 `size/quality/style`；`ratio` 只在页面状态与历史恢复中使用，没有直接提交给上游。
- 种子受 `seed_control` 控制，开启且输入后通过 `seed` 字段提交。
- `response_format` 固定为 `url`。

数量逻辑：

- 数量不是提交给上游的 `n`，而是前端循环调用接口多次。
- 单次 payload 不包含 `n`，因此后端 `validateWorkspaceImageGenerationRequest()` 的 `N` 校验通常只看到默认 1。
- 前端用渠道 `max_batch_size` 限制循环次数。后端也有 `N` 限制，但当前循环调用方式主要靠前端限制。
- 任务中心会为每次调用生成一条独立任务记录。
- 批量提交时会打开进度弹窗，显示当前生成进度、成功数量和失败数量；某一次失败不会中断后续提交，全部失败时弹窗进入错误态。

任务中心逻辑：

- 工作台图片记录来源为 `workspace_image`，提交来源为 `workspace`。
- API 图片生成 `/v1/images/generations` 现在被 `RelayAPIImageGeneration` 包装，仍调用原 `Relay(OpenAIImage)`，同时记录来源为 `api_image`。
- 图片结果和参考图会进入任务中心 detail，之后异步本地化下载到 `data/task-center`；下载失败会保留原链接。

风险点：

- 图片的“比例预设”目前不会直接进入上游请求，只作为 UI/历史参数存在。若某些上游需要 `ratio/aspect_ratio`，需要后续适配。
- 图片数量是多次提交，不是 `n`。这符合你的要求，但后端无法通过单个请求识别“批量次数”，只能看到多条单任务。

## 视频生成页面

请求入口：`web/default/src/features/workspace-generation/index.tsx`

接口入口：`/api/workspace/video/generations`

后端入口：`controller/workspace_video.go`

中转路径：`GenerateWorkspaceVideo` -> `middleware.Distribute()` -> `RelayTask()` -> `relay.RelayTaskSubmit()` -> `service.PreConsumeBilling()` / `service.SettleBilling()` / `service.LogTaskConsumption()`

参数逻辑：

- 模型分组来自 `/api/workspace/video/models`。
- 模型渠道显示优先级为 `model_alias/display_name/model`，提交给上游仍使用真实 `model`。
- 提示词 `prompt` 必填。
- 首帧图优先通过 `image` 字段提交；没有首帧图时，参考图可作为 `image` 兜底。
- 参考图、参考图数组、尾帧图进入 `metadata.reference_image/reference_images/last_frame_image`。
- 分辨率通过 `size` 提交。
- 时长通过 `duration` 提交，前端把 `5s` 解析为 `5`。
- 比例、帧率、风格、质量、负面提示词、音频、种子通过 `metadata` 提交。

数量逻辑：

- 数量不是提交给上游的生成数量，而是前端循环调用多次接口。
- 每次提交都会进入原 Task 体系和任务中心，任务中心显示每一条任务。
- 前端用视频渠道 `max_batch_size` 限制提交次数。
- 批量提交时会打开进度弹窗，显示当前提交进度、成功提交数量和失败数量；某一次提交失败不会中断后续提交，成功提交的视频任务仍继续在后台异步生成。

任务中心逻辑：

- 工作台视频通过 `workspace_source = workspace_video` 标记。
- `Task.Insert()`、`Task.UpdateWithStatus()` 会同步到任务中心。
- 完成后 `FinishTime` 会用于任务中心完成时间与使用日志耗时更新。
- 生成视频、参考图会异步下载到本地 `data/task-center`；下载失败保留原链接。

风险点：

- 视频工作台后端已校验模型是否存在于启用的视频渠道、分类是否禁用、功能控制是否允许；但“批量次数”仍按前端多次单任务提交实现，后端只能拦截单次请求中显式伪造的 `n/count/batch_size`，不能阻止用户连续发起多个独立请求。
- 视频“相机运动”已在 `camera_control` 开启时提交到 `metadata.camera_movement`，关闭时前端清空且后端拦截相关 metadata 字段。
- `GetWorkspaceVideoAvailableModels()` 复用了聊天可用模型列表。管理页能选择“可用模型”，但不区分模型是否真的支持视频，仍需管理员正确配置。

## 原版 API 中转影响

文本/聊天 API：

- 主入口 `controller.Relay()` 未被改成工作台专用逻辑。
- 扣费仍是 `ModelPriceHelper()` -> `PreConsumeBilling()` -> 各 relay handler -> `PostTextConsumeQuota()` -> `SettleBilling()`。
- 未发现工作台页面改动直接改变普通聊天 API 的路由或扣费计算。

图片 API：

- `/v1/images/generations` 现在进入 `RelayAPIImageGeneration()`，这个函数读取请求体、重置 body、调用原 `Relay(OpenAIImage)`，再把响应写入任务中心。
- 正常响应体仍由原 relay 写出，任务中心记录是附加行为。
- 影响：多一次请求体读取、响应体缓存、任务中心 DB 写入和异步资源下载。高并发图片 API 会增加轻微 I/O 与数据库压力。

视频 API：

- 原异步 Task 体系仍负责提交、轮询、扣费、退款、差额结算。
- 新增任务中心同步来自 Task 插入和状态更新，不改变上游提交方式。
- `source = workspace_video` 仅工作台视频使用，普通 API 视频任务仍显示为 API 来源。

任务中心资源访问：

- `/api/task-center-assets/*path` 当前没有登录校验，知道本地资源 URL 就可访问。此前你要求去掉权限限制，这符合需求，但它是一个明确的公开访问面。

## 扣费逻辑

同步文本/图片扣费：

- 文本、聊天、图片最终进入 `service.PostTextConsumeQuota()`。
- 图片 `n` 通过 `OtherRatios["n"]` 只计算一次，避免重复扣费。
- 工作台图片使用临时 token 上下文走原 relay 扣费，不单独写新的扣费算法。
- API 图片任务中心费用从使用日志按 `request_id` 反查 quota。
- 工作台图片失败时，如果 relay 没有成功写入对应使用日志，任务中心记录的 `cost` 为 0，不会额外扣费。

异步视频扣费：

- `RelayTaskSubmit()` 使用 `ModelPriceHelperPerCall()` 估算价格。
- 提交时 `ForcePreConsume = true` 进行预扣。
- 成功提交后 `SettleBilling()`、`LogTaskConsumption()` 记录任务日志和使用日志。
- 后续轮询成功/失败时，`RecalculateTaskQuota()` 或 `RefundTaskQuota()` 做差额结算或退款。
- 聚焦测试 `TestRefundTaskQuota`、`TestRecalculateTaskQuota` 已通过。
- 视频任务提交失败会走 relay 的预扣退款；异步轮询后失败会在 `controller/task_video.go` 的失败分支按原 `task.Quota` 返还，且原状态已经是失败时会跳过重复退款。

失败数据验证：

- 测试标记：`synthetic_failed_1780485366`。
- 失败图片任务：`img_synthetic_failed_1780485366`，`task_centers.cost = 0`。
- 失败视频任务：`task_video_synthetic_failed_1780485366`，`task_centers.cost = 0`，`tasks.quota = 0`。
- 写入前 admin：`quota=20981286`，`used_quota=79018715`。
- 写入后 admin：`quota=20981286`，`used_quota=79018715`。
- 匹配测试标记的消费日志新增数：0。

结论：

- 当前改动没有发现直接改坏原扣费主链路。
- 任务中心是新增记录层，费用展示依赖原使用日志和 Task quota。
- 本次导入的失败图片/视频测试数据没有触发扣费；真实失败任务按当前代码路径也应表现为未扣费或已退款。
