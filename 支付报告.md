# 支付逻辑报告

## 1. 当前支付网关状态

当前代码中仍存在以下支付网关与接口：

- Epay：普通在线充值，当前支付宝、微信支付都走这套逻辑。
- Stripe：独立 Stripe Checkout 支付。
- Creem：代码和接口仍存在。
- Waffo / Waffo Pancake：代码和接口仍存在。
- 虎皮椒 / XunhuPay：当前代码中未发现实际控制器、配置项或路由入口。

结论：当前微信、支付宝并不是独立虎皮椒支付逻辑，而是 Epay 支付方式中的两个 `payment_method`：

```text
alipay = 支付宝
wxpay  = 微信支付
```

## 2. 前台展示逻辑

用户进入钱包充值页面后，前端会请求：

```text
GET /api/user/topup/info
```

后端返回主要数据：

```text
enable_online_topup
enable_stripe_topup
enable_creem_topup
enable_waffo_topup
enable_waffo_pancake_topup
pay_methods
min_topup
stripe_min_topup
amount_options
discount
topup_link
```

前端根据 `pay_methods` 渲染普通支付方式按钮，例如支付宝、微信、Stripe 等。

如果后台支付合规未确认，后端会将 `pay_methods` 置空，用户前台无法看到可用支付方式。

## 3. 微信和支付宝执行逻辑

微信和支付宝当前都属于 Epay 普通在线充值。

用户点击支付方式后，前端先计算实际付款金额：

```text
POST /api/user/amount
```

提交参数：

```json
{
  "amount": 充值数量
}
```

后端根据以下配置计算实际支付金额：

- `Price`
- 用户分组充值倍率
- 充值金额折扣 `payment_setting.amount_discount`
- 当前额度展示类型

用户确认支付后，前端请求：

```text
POST /api/user/pay
```

提交参数：

```json
{
  "amount": 充值数量,
  "payment_method": "alipay 或 wxpay"
}
```

后端执行流程：

1. 校验充值数量是否低于最低充值。
2. 获取用户分组。
3. 计算实际付款金额。
4. 校验 `payment_method` 是否存在于后台 `PayMethods` 配置。
5. 使用 Epay 配置初始化支付客户端。
6. 创建本地充值订单，状态为 `pending`。
7. 调用 Epay SDK 生成支付表单参数。
8. 返回支付网关地址和参数给前端。
9. 前端创建隐藏表单并提交到 Epay 网关。

## 4. Epay 回调与到账逻辑

Epay 回调接口：

```text
GET  /api/user/epay/notify
POST /api/user/epay/notify
```

后端处理流程：

1. 检查 Epay 是否启用。
2. 解析 GET Query 或 POST Form 参数。
3. 使用 Epay SDK 验签。
4. 判断交易状态是否为成功。
5. 根据回调订单号查询本地 `top_ups` 订单。
6. 校验订单支付网关必须是 `epay`。
7. 只有订单状态为 `pending` 时才会更新。
8. 更新订单状态为 `success`。
9. 给用户增加额度。
10. 写入充值日志。

代码中使用了订单锁 `LockOrder(tradeNo)`，用于降低重复回调并发处理导致重复到账的风险。

## 5. Stripe 执行逻辑

Stripe 不走 Epay 表单。

前端请求金额计算：

```text
POST /api/user/stripe/amount
```

用户确认后请求：

```text
POST /api/user/stripe/pay
```

提交参数：

```json
{
  "amount": 充值数量,
  "payment_method": "stripe"
}
```

后端执行流程：

1. 校验 `payment_method` 必须为 `stripe`。
2. 校验最低充值和最大充值限制。
3. 根据 Stripe 单价和用户分组计算实际金额。
4. 创建 Stripe Checkout Session。
5. 创建本地 `top_ups` 订单，状态为 `pending`。
6. 返回 `pay_link`。
7. 前端打开 Stripe 支付页面。

Stripe 回调接口：

```text
POST /api/stripe/webhook
```

后端校验 `Stripe-Signature`，处理这些事件：

```text
checkout.session.completed
checkout.session.expired
checkout.session.async_payment_succeeded
checkout.session.async_payment_failed
```

支付成功后调用充值逻辑，更新订单状态并增加用户额度。

## 6. 后台配置项

支付网关配置位于：

```text
系统设置 -> 计费与支付 -> 支付网关
```

Epay 相关配置：

```text
PayAddress
EpayId
EpayKey
PayMethods
Price
MinTopUp
CustomCallbackAddress
```

Stripe 相关配置：

```text
StripeApiSecret
StripeWebhookSecret
StripePriceId
StripeUnitPrice
StripeMinTopUp
StripePromotionCodesEnabled
```

普通支付方式配置示例：

```json
[
  {
    "name": "支付宝",
    "type": "alipay",
    "color": "rgba(var(--semi-blue-5), 1)"
  },
  {
    "name": "微信",
    "type": "wxpay",
    "color": "rgba(var(--semi-green-5), 1)"
  }
]
```

其中 `type` 会作为 `payment_method` 传给后端，再由后端传给 Epay。

## 7. 当前结论

当前代码没有发现虎皮椒独立支付实现。

如果需要虎皮椒微信和虎皮椒支付宝两个独立通道，需要新增或恢复专门的 XunhuPay 网关逻辑，并至少补齐：

- 后端配置项：微信 APPID/Secret、支付宝 APPID/Secret。
- 后端支付接口：区分微信和支付宝虎皮椒通道。
- 后端回调接口：验签、查单、幂等到账。
- 前端支付方式：`xunhupay_wechat`、`xunhupay_alipay`。
- 后台支付网关页面：分别保存微信和支付宝配置。

当前版本中，支付宝和微信只能作为 Epay 的支付类型使用，不能分别配置独立 APPID 和密钥。
