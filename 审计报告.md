## 功能影响分析

### 1. 工作区图片生成

影响等级：中高

变更内容：

- 新增图片估算接口：`POST /api/workspace/image/estimate`
- 图片生成请求会根据渠道配置的 `field_mappings` 透传参数。
- 图片生成页面按钮展示预计消耗。
- 图片提交行为由“等待生成结果后成功”改为“提交即成功”。
- 参考图预览改为自适应缩放显示。

正向影响：

- 图片页面提交体验更快。
- 可按渠道配置字段名，例如将尺寸映射为 `Size`、`size` 或其他上游要求字段。
- 功能控制关闭时，对应参数不会传递给上游。
- 预计消耗展示可以降低用户提交前的不确定性。

潜在风险：

- 已修复：图片实际生成提交已改为按数量循环提交单张请求，实际调用 `generateWorkspaceImage(buildImagePayload())` 时默认不携带 `n`，避免不支持 `n` 的上游报错。`n` 仅保留在估算请求中，用于提交前计算总预计消耗，不触发真实生成。
- 提交即成功后，真实生成失败会变成异步失败提示，用户可能先看到成功 toast，稍后再看到失败。
- 字段映射配置错误会导致上游收不到参数或收到错误字段。
- 估算消耗与最终扣费可能存在差异，特别是上游任务失败、退款、后置结算或动态倍率场景。

已完成修复：

1. 图片数量控制仍保留在前端显示和价格估算中，但实际生成按数量循环创建多个单张任务。
2. 每个实际生成请求默认不带 `n` 字段，避免把批量能力强行交给不支持的上游。
3. 提交结果按成功/失败聚合展示，失败只影响当前单次任务，不阻断已经成功提交的任务。
4. 对支持 `n` 的上游，如后续需要优化吞吐，可再增加渠道级“允许批量透传”开关；当前默认策略保持循环提交，兼容性最高。

### 2. 工作区视频生成

影响等级：中高

变更内容：

- 新增视频估算接口：`POST /api/workspace/video/estimate`
- 视频渠道新增字段映射配置。
- 视频渠道新增风格预设、质量预设、运镜预设。
- 视频生成请求会按映射生成顶层字段或 `metadata` 字段。
- 视频按钮展示预计消耗。

正向影响：

- 更适配 VEO、Gemini、Vertex、Sora 等不同上游字段结构。
- 能配置中文预设显示，同时保留传给上游的实际值。
- 功能控制关闭时，对应参数不会传递。

潜在风险：

- 已修复部分兼容问题：视频校验已增加旧请求兼容逻辑。渠道关闭首帧图但开启参考图时，若请求只带顶层 `image`，且没有显式首帧/尾帧/多参考图字段，会按旧参考图场景兼容处理，不再直接误判为“不支持首帧图”。
- 首帧图、参考图、尾帧图之间的判定仍比旧逻辑复杂。顶层 `image` 在不同上游中可能代表首帧图，也可能代表参考图；如果同时配置了复杂映射，仍需要按渠道逐一验证。
- 映射到 `metadata.xxx` 与顶层字段的策略依赖配置正确性。不同视频适配器读取字段的位置不同，VEO、Gemini、Vertex、Sora 等上游对 `duration`、`seconds`、`size`、`resolution`、`aspect_ratio`、`image`、`images` 的识别规则需要分别验证。
- 功能控制关闭时会阻止对应参数传递，但如果映射字段配置不清晰，可能出现“前端看起来填写了参数，后端校验或上游适配器没有按预期读取”的情况。
- 估算逻辑依赖任务适配器 `EstimateBilling`，适合作为提交前预计消耗展示；最终扣费仍可能因实际任务结果、轮询结算、失败退款、按秒结算或上游返回值而不同。

已完成修复与剩余计划：

1. 已完成：对 `image` 字段增加旧参考图兼容判断，避免关闭首帧图但开启参考图时的误拒。
2. 已完成：视频功能控制校验会结合字段映射检查顶层字段和 `metadata.xxx` 字段，避免关闭功能后仍透传对应参数。
3. 剩余：为每个主流视频上游建立字段映射约定表，明确哪些参数应走顶层字段，哪些参数应走 `metadata.xxx`。
4. 剩余：在视频渠道管理页面增加配置提示或默认模板，降低 VEO/Gemini/Vertex/Sora 等不同上游的映射配置错误率。
5. 剩余：增加人工验证矩阵：文本生视频、首帧图、参考图、尾帧图、首尾帧组合、时长、比例、分辨率、质量、运镜、种子。
6. 消耗展示继续使用“预计”措辞；估算失败时保持按钮可用但显示“暂无法估算”，并在日志或接口响应中保留失败原因，方便排查价格或渠道配置。

## 数据库兼容性审计

新增字段：

- `workspace_image_channels.field_mappings`
- `workspace_video_channels.field_mappings`
- `workspace_video_channels.camera_movement_presets`

代码使用 GORM `AutoMigrate`，并且迁移列表中已包含 `WorkspaceImageChannel` 和 `WorkspaceVideoChannel`。这三个新增字段都是 `TEXT` 类型的追加字段，不涉及改列、删列或数据库专有类型；在当前代码路径正常执行迁移的前提下，SQLite、MySQL、PostgreSQL 已有数据库缺少这些列时可以自动补齐。

风险点：

- 如果生产环境启动时没有执行 `migrateDB()` / `migrateDBFast()`，或者部署流程关闭了自动迁移，已有数据库不会自动补列。
- 如果数据库账号没有 `ALTER TABLE` / DDL 权限，自动迁移会失败，渠道保存或更新时可能出现字段不存在错误。
- 如果线上有多实例部署，建议先完成数据库迁移，再滚动更新应用。
- GORM `AutoMigrate` 适合本次“新增文本列”的场景，但不应被理解为能安全处理所有历史结构变更；复杂改列仍需要专门迁移脚本。

上线建议：

1. 上线前确认数据库已存在上述列，或确认应用启动会执行自动迁移且数据库账号具备 DDL 权限。
2. 至少在测试环境保存一次图片渠道和视频渠道。
3. 如生产关闭自动迁移，应准备手动 SQL 迁移脚本，至少补齐：
   - `workspace_image_channels.field_mappings TEXT`
   - `workspace_video_channels.field_mappings TEXT`
   - `workspace_video_channels.camera_movement_presets TEXT`
4. 多实例部署时，优先单独完成数据库迁移，再滚动更新应用实例，避免新旧实例同时读写新增字段时出现不一致。
