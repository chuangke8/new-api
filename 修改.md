## 一、聊天工作台

### 1. 模型选择后自动匹配对应分组

问题：

- 聊天页面选择模型后，如果当前仍使用旧的模型分组，可能请求到错误分组。
- 曾出现类似提示：`分组 default 下模型 gpt-5.5 无可用渠道`。
- 你明确要求：根据模型获取分组，而不是根据分组选择模型。

需求：

- 用户在聊天页面选择模型后，系统应自动选择该模型对应的可用分组。
- 如果同一个模型在多个分组下可用，应优先选择可用渠道所在分组。
- 不应让用户因为残留的旧分组导致请求失败。
- 需要避免影响用户自己的用户组、额度组、权限组逻辑。

建议实现点：

- 聊天前端模型选择逻辑中，根据模型元数据或可用渠道信息同步更新请求分组。
- 提交聊天请求前再次校验模型和分组是否匹配。
- 如果无法确定分组，给出中文提示，而不是直接使用旧分组。

### 2. 隐藏聊天页面的模型分组选择

问题：

- 聊天页面曾显示类似 `VIP`、`Default`、`Codex` 这类模型分组。
- 你要求隐藏的是模型分组，不是用户分组。

需求：

- 聊天页面不再让普通用户手动切换模型分组。
- 用户只需要选择模型。
- 分组应由系统根据模型自动判断。

注意：

- 不要误隐藏用户自己的权限分组或账户分组。
- 管理端渠道、模型、分组配置仍应保留。

### 3. 聊天页面增加模型分类筛选

需求：

- 在原来模型分组选择所在位置，增加“模型分类”。
- 这个分类来自聊天渠道管理里的模型分类。
- 用户可以按分类筛选聊天模型。

建议：

- 分类用于前端筛选展示，不应直接当作请求分组传给中转接口。
- 选择分类后，仅改变模型列表展示。
- 选择模型后，再由模型自动匹配实际请求分组。

## 二、图片生成工作台

### 1. 图片生成提交后立即成功

问题：

- 图片生成页面提交后，需要等待上游生成完成才返回成功，耗时过长。
- 你要求：提交无需等待，提交就算成功，然后后台继续生成。

需求：

- 用户点击生成图片后，前端应尽快得到提交成功反馈。
- 后端应创建任务并进入后台处理。
- 前端不应阻塞等待上游最终图片生成完成。
- 后续结果通过任务中心、轮询或任务详情查看。

注意：

- 任务提交成功不等于生成成功。
- 前端文案应区分“已提交”和“已生成”。
- 后台任务失败时，需要在任务状态中体现失败原因。

### 2. 多张图片数量逻辑改为循环提交

问题：

- 如果某些上游不支持 `n` 批量参数，直接透传数量可能导致上游报错。
- 你明确要求：数量改为循环提交，不是把批量数量递交给上游。

需求：

- 图片数量字段仍保留在前端。
- 后端根据数量循环创建或提交多次任务。
- 不把前端数量字段直接作为 `n` 传给不支持的上游。
- 每次上游请求应是单次生成或渠道支持的安全请求。

建议：

- 数量字段作为本系统任务数量控制。
- 上游请求中是否包含数量字段，由渠道功能控制决定。
- 默认更安全的策略是系统循环提交。

### 3. 图片输入组件 UI 美化

需求：

- 优化图片生成页面输入区域的视觉和交互。
- 参考图上传后应自动缩放预览。
- 预览不应撑破布局。
- 表单控件应更清晰，适合反复使用。

建议：

- 参考图、首帧图、尾帧图等图片预览使用固定容器比例。
- 图片较大时自动等比缩放。
- 长文件名、长 URL 不应导致布局溢出。

### 4. 图片生成按钮显示本次消耗金额

需求：

- 图片生成页面生成按钮上展示本次预计消耗金额。
- 文案使用左右排列，不要上下排列。
- 如果无法估算，需要检查原因并尽量给出可读提示。

建议：

- 按模型、渠道、数量、分组、价格配置估算。
- 当价格缺失时显示“暂无法估算消耗”，并在调试中能定位缺失项。
- 按钮布局应保持稳定，不因金额变化抖动。

## 三、图片渠道管理

### 1. 图片渠道参数透传

需求：

- 图片渠道管理中，除数量字段在前端作为系统循环控制外，其他开启的字段需要完整传递给上游。
- 如果功能控制将某项功能关闭，则该字段不应提交给上游。

涉及字段示例：

- 尺寸
- 比例
- 质量
- 风格
- 参考图
- 首帧图
- 尾帧图
- 负面提示词
- 其他渠道自定义参数

规则：

- 功能开启：允许前端填写，并按映射字段传给上游。
- 功能关闭：前端隐藏或禁用，并且后端不透传。
- 数量字段：优先作为本系统循环提交控制，不直接递交给上游。

### 2. 图片渠道字段映射

需求：

- 图片渠道新建和编辑页面增加字段映射能力。
- 例如本系统参数名是 `sizes`，上游要求字段名是 `abcd`，配置映射后应按 `abcd` 传给上游。

需求细节：

- 每个可控参数应能配置上游字段名。
- 字段映射为空时使用默认字段名。
- 字段映射后，请求上游时使用映射后的字段名。
- 映射只改变上游请求字段，不改变前端显示名称。

示例：

- 前端参数：`sizes`
- 映射字段：`abcd`
- 上游请求：`{ "abcd": "1024x1024" }`

### 3. 图片渠道新建页面显示问题

需求：

- 图片渠道管理中新建渠道页面，全屏或较小屏幕时显示不全，需要处理。
- 模型分类选择后，应显示模型分类名称，而不是只显示 ID。

建议：

- 弹窗或抽屉内容区域支持滚动。
- 选择器回显使用分类名称。
- 保存时仍提交分类 ID。

## 五、视频和图片通用字段映射规则

### 1. 字段映射优先级

建议规则：

- 如果配置了字段映射，使用映射字段名提交给上游。
- 如果没有配置映射，使用系统默认字段名。

- 管理端可以配置中文名称。

- 显示名称：`高清`
- 实际值：`high`
- 映射字段：`quality`
- 上游请求：`{ "quality": "high" }`

### 3. 参考图、首帧图、尾帧图处理

需求：

- 参考图、首帧图、尾帧图之间需要明确区分。
- 不应把普通参考图误判成首帧图。
- 功能关闭时，不提交对应图片字段。
- 功能开启时，根据渠道映射传给上游。

建议：

- 内部字段建议清晰区分：
  - `reference_images`
  - `first_frame_image`
  - `last_frame_image`
- 上游字段名通过映射配置决定。
